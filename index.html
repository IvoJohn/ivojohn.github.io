Time series analysis

Abstract

In this project I performed analysis and plotting of a time series of energy consumption. Data was obtained from hourly measurements of energy consumption done by PJM Interconnection and published on kaggle.
I explored and plotted the data with interactive plots done in matplotlib and then process them to a form suitable for network input. I tried different approaches, from the simplest, one neuron network to combination of 2D CNN with LSTM. The best prediction obtained mean squared error of 6.2e-5.

Data exploration

The data is consisted of a single feature time-series registered every hour. The registered value is an energy consumption in megawats. There are no missing values and the total amount of them is equal to 145 366. The beginning of measurements is 01.01.2002 01:00 and the end is 03.08.2018 00:00. The mean value of energy consumption is 32080.22MW, standard deviation is 6464.01MW, minimal value is 14544.0MW, maximal value is 62009.0MW Above we can see histogram with 20-bins. The population is right-skewed. Below there is a box plot showing both the trend and seasonality of the data. There is no observable trend in the data but there is a visible seasonality on the yearly-basis. After the closer inspection we can also observe sesonality on the daily-basis. In the code there is more plots and the one below is interactive, enabling to show seasonality during year, week and day of the week with possibility of choosing any avaiable day and year. In the code there is also a plot for multiplicative and additive decomposition. Above we have a plot of moving average with window set to 31 days. This plot is also interactive, with possibility to change the size of the window from 10 to 88 days. With the plot above we can also observe clear seasonality of the data.

Data preparation

In this step I prepared the data to suit the format needed by neural networks architectures. Buf before that I check the stationarity of the data. This was done with Augmented Dickey-Fuller test.
Augmented Dickey-Fuller test is a type of unit root test which performs autoregression with the null hypothesis of a time-series being non-stationary. The P-value of this test was equal to 2.02e-30 which is way below the threshold value of 5% so I abondoned further processing data to eliminate its non-stationarity since it was virtualy non-existing. The data was divided into training, validation and test sets, scalled with a scaller fitted only to training data to avoid lekage of any information to validation and test sets and as the last step splitted into X and y with window size of 30 and shift of 1. Sequence length was set to 20.

Forecasting

Single layer model
As the reference point I started from simples possible model, with only one layer and one neuron. Due to small computational cost I decided to find the most optimal learning rate with Learning Rate Scheduler and plot of learning rate vs epoch. The most optimal value was equal to 2e-3 so it was set as the learning rate for a Adam optimizer in the final model. I used Huber loss as it is more resistant to outliers (like the one presented in data, which corresponds to the hurracane which takes place during the measurement period, resulting in decreass of energy consumption) and as a metric I set Mean Standard Error. The model was trained for 120 epochs and still have not reached platau nor the validation loss started increasing or fluctuating. This shows that the model could be trained even longer. The Mean Squared Error calculated on previously unseen test set was equal to 3.77e-4.
Multi layer model
The next model was still composed of only fully connected layers But this time I applied many more neurons to each of the layers and also added Dropout equal to 20% between each of them. The model clearly overfitted which is visible both on the discrapency of values between train loss and validation loss and also on the plots. This could be prevented with fewer layers (so simpler model) or with bigger dropout value between layers. The value of MSE on test set was equal to 1e-3.
LSTM
Since we are dealing with time series, the most succesful architecture should be the one designed to learn from time data. And that was the case. The applied network was constructed from two unilateral LSTM layers, one dense layer with relu activation and dropout of 20% before the output layer. I trained the network for 50 epochs and still did not reach plateau. But the most prominent fact was that the validation loss was a full order of magnitude smaller than training loss which clearly shows lack of overfitting, probably even bigger model could be used to this data. Also the value of MSE on test set was the smalles among applied architectures: 6.2e-5.

LSTM with Convolution
This architecture turned out to be the worst among applied ones. After training for 50 epochs it clearly overfitted the data with training loss an order of magnitute smaller than in the case of only LSTM layers and the value of validation loss was on the contrary order of magnitute bigger. The value of MSE on test data was the worst and equal to 1.3e-3.
Discussion
The most effective network in this case turned out to be the one constructed with two LSTM layers, the quality of predictions was high. It could be even higher if the network would be trained for a longer time or maybe slightly bigger. The application of a correct architecture enables to reduce the Mean Squared Error on previously unseen test set by the two orders of magnitude. Another approaches which could be applied would be forecasting the data after removing sesonality or with other recurrent architectures.

Chest X-ray classification

Abstract

In this project I used two convolutional neural networks to detect pneumonia on chest x-rays. The accuracy on training set exceeded 96%. The working networks were placed inside graphical user interface made with PyQt5. The interface provides easy and intuitlional way of interacting with backend networks.

Interface

The main functionality of the program is provided after uploading image with a button opening file menager. The first network is responsible for classifying if the uploaded image presents chest x-ray or not. In the letter case an error window shows up and asks for uplouding different image. If the image is correct the result of classification with assigned certainty of it is shown. Apart from classifying images the interface offers possibility to read the short documentation of the backend solutions and to remaind oneselfs that this is not anything close to a clinicaly ready solution. Both pieces of information are avaiable after clicking on a designeted buttons.

Backend

The networks which provides core functionality of the program were written in Tensorflow with Keras. Both of them have the same architecture with pretrained VGG16 followed by Flatten layer, two dropouts and two Dense layers. The usage of pretrained network exceeded all other applied approaches and reached over 93% of accuracy as early as in the second epoch. The network classifying non-pneumonia vs pneumonia images was trained for 15 epochs. The network classifying presence of a chest x-ray on the uploaded image was trained for only one epochs due to custom callback which interupted training after exceeding 97% of accuracy. In both cases image data generators were used to provide more variability in the training set. In the case of pneumonia detection the augmentation was rather small due to the assumption that x-ray images have a standarized procedure of performing and there should not be a big variance between their rotation etc. Also in the case of pneumonia detection I applied weights to the classes since the number of positive examples was almost three times larger. Due to only two classes weights were calculated by hand, without usage of scikit-learn tools.

Further development

The further developement of the project could include adding another classes to distingish between, like separating bacterial from viral pneumonia, or lobar from intersitial pneumonia. Another functionality could be visualisation of convolutional filters activation in the network. Probably the most interesing solution would be an automatic description generator, but this is limited by the amount of data avaiable.
